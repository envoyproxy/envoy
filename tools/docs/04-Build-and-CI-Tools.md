# Build and CI Tools

This document covers the tools that support Envoy's Bazel build system, continuous integration, Docker-based testing, and build profiling.

---

## Overview

Envoy uses Bazel as its build system. The tools in this category provide infrastructure for running builds and tests in various environments, generating compilation databases for IDE support, profiling build performance, and managing CI workflows.

```
tools/
├── base/                              # Shared Python tooling foundation
│   ├── envoy_python.bzl               # envoy_py_binary, envoy_jinja_env macros
│   ├── requirements.in / .txt         # Base Python dependencies
│   └── BUILD
├── ci/                                # CI reporting
│   └── BUILD                          # Defines report py_console_script_binary
├── toolchain/                         # Toolchain detection
│   ├── detect.cc                      # Compile-time toolchain detection
│   └── BUILD
├── cat/                               # Minimal cat-style utility
│   └── BUILD
├── jq/                                # JQ processing utility
│   └── BUILD                          # Defines jqcat
├── tarball/                           # Zstd tarball unpacker
│   └── BUILD                          # Defines unpack
├── bazel-test-docker.sh               # Run Bazel tests in Docker
├── docker_wrapper.sh                  # Docker wrapper for test invocations
├── build_profile.py                   # Build time profiler
├── gen_compilation_database.py        # Generate compile_commands.json
├── run_command.py                     # Shell command runner library
└── path_fix.sh                        # Fix Bazel paths in compiler errors
```

---

## `tools/base/` — Shared Python Foundation

The foundation for all Python tools in the repository.

### `envoy_python.bzl`

Defines the core Bazel macros:

- **`envoy_py_binary`** — Wraps a Python script as a runnable Bazel target with automatic test generation. For every `envoy_py_binary` named `tools.foo.bar`, a corresponding `pytest_bar` test target is created automatically.
- **`envoy_jinja_env`** — Sets up Jinja2 template environments for tools that generate templated output.

### `requirements.in` / `requirements.txt`

Base Python dependencies shared across all tools. The `.in` file lists direct dependencies; the `.txt` file is the fully resolved, hash-pinned version generated by `pip-compile`.

---

## `tools/ci/` — CI Reporting

Provides the `report` tool for CI pipelines. This is a `py_console_script_binary` that generates structured reports from CI job outputs, used by the Envoy CI system to summarize test results and build status.

---

## `tools/bazel-test-docker.sh` — Docker Test Runner

Runs a Bazel test target inside a privileged Docker container. Useful for tests that require elevated permissions (network namespaces, raw sockets, etc.) or need a consistent Linux environment regardless of the host OS.

**Usage:**

```bash
# Run a test target in Docker
tools/bazel-test-docker.sh //test/integration:integration_test
```

The script handles:
- Pulling or building the appropriate Docker image
- Mounting the workspace into the container
- Forwarding Bazel cache for efficient rebuilds
- Running the test with `--privileged` for tests requiring system access

---

## `tools/docker_wrapper.sh` — Docker Wrapper

A more general Docker wrapper that runs arbitrary test invocations inside Docker containers. Can operate against both local and remote Docker daemons.

---

## `tools/build_profile.py` — Build Time Profiler

Profiles Bazel build times by parsing dependency logs (`*.dep.log`) produced by `recipe_wrapper.sh`. Identifies slow targets and bottlenecks in the build graph.

**Usage:**

```bash
python3 tools/build_profile.py path/to/target.dep.log
```

**Output:** A sorted breakdown of build time by target, helping identify which targets to optimize for faster builds.

---

## `tools/gen_compilation_database.py` — Compilation Database Generator

Generates `compile_commands.json` at the workspace root. This file is consumed by:

- **clangd** — for code completion, go-to-definition, and diagnostics
- **VS Code** / **Cursor** — via the clangd extension
- **clang-tidy** — for static analysis
- Other tools that understand the [JSON Compilation Database](https://clang.llvm.org/docs/JSONCompilationDatabase.html) format

**Usage:**

```bash
# Generate for the full project
python3 tools/gen_compilation_database.py

# Or via CI script (recommended — also generates proto dependencies)
ci/do_ci.sh refresh_compdb
```

The `ci/do_ci.sh refresh_compdb` variant is preferred because it also builds protobuf-generated code and external dependencies, ensuring clangd can resolve all symbols.

---

## `tools/run_command.py` — Shell Command Library

A Python library (not a standalone tool) that provides a clean interface for running shell commands and capturing their stdout/stderr. Used internally by several other tools.

**Key function:** `run_command(cmd)` — executes a command and returns its output.

---

## `tools/toolchain/` — Toolchain Detection

### `detect.cc`

A small C++ file that, when compiled, reports information about the active toolchain (compiler version, target architecture, etc.). Used by build infrastructure to adapt behavior based on the compilation environment.

---

## `tools/path_fix.sh` — Bazel Path Translator

Translates Bazel sandbox paths in compiler error messages to real filesystem paths. Without this, error messages from Bazel builds reference paths like `bazel-out/k8-fastbuild/bin/...` which can't be clicked in IDE terminals.

**Usage:**

```bash
# Pipe compiler output through path_fix
bazel build //source/exe:envoy-static 2>&1 | tools/path_fix.sh
```

---

## Supporting Build Utilities

### `tools/cat/`

A minimal `cat`-style Bazel utility. Used internally by build rules that need to concatenate file contents during the build.

### `tools/jq/`

Defines the `jqcat` target — a JQ processing utility used by dependency and CVE tools for filtering and transforming JSON data.

### `tools/tarball/`

Defines the `unpack` target for extracting zstd-compressed tarballs. Used by build rules that consume prebuilt artifacts.
