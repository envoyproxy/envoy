JwksAsyncFetcher ran its JWKS refetch timer on the main dispatcher independent of any connection. The lifecycle of JwksAsyncFetcher ultimately is tied to the filter chain and on an in-place filter chain update (e.g. a config change), the replaced filter chain was kept alive for its drain period (600s) but its fetcher kept fetching, so every config update "leaked" another parallel fetch loop against the same JWKS endpoint until drain completed. The fetcher now checks the owning filter chain's drain decision: it bails out of fetch() and stops re-arming the refetch timer once the chain (or its listener/server) is draining, leaving one active fetch loop per provider regardless of config churn.
